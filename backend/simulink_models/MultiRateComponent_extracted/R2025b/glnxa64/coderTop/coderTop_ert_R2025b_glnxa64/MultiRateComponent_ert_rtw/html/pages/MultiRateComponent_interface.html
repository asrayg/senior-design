1. Introduction [show]

From a component model, the code generator produces code that has a well-defined programming interface for interacting with other platform environment software. Depending on your platform selection and service configurations, the code generator produces:

  • A collection of callable entry-point functions that align with platform scheduler service calls
  • Calls to platform service functions that align with the service programming interface
  • Generated function code that honors data communication methods that platform services use
This figure shows the interaction that occurs between a component and platform services in a platform execution environment. The arrows indicate the direction of function interface calls. The platform function scheduler calls component entry-point functions: Initialize, F1, F2, and Terminate. The component calls platform services to read input, write output, transfer data between callable functions, and get delta time.

2. Function Interfaces [hide]

Target platform software initiates execution of code generated from a model by calling generated entry-point functions. The entry-point functions that the code generator produces for a model depends on the modeling style and periodic rates represented in the model. For more information, see Periodic and Aperiodic Function Interfaces and Startup, Reset, and Shutdown Function Interfaces.

2.1 Initialize Functions

Initialize entry-point functions implement startup behavior. In a model, Initialize Function blocks represent startup behavior explicitly.

MultiRateComponent_initialize

Initialization entry point of generated code
#include "MultiRateComponent.h" void MultiRateComponent_initialize(void)
Calls services:
  • Receiver: get_MultiRateComponent_initialize_InNVM #include "services.h" real_T get_MultiRateComponent_initialize_InNVM(void)

    Data communication method: Outside execution
    Target platform communicates data before or after generated code executes.

2.3 Periodic Functions

Periodic entry-point functions implement model behavior that occurs at a fixed sampling rate. For a rate-based model, the code generator produces a periodic entry-point function for each rate used in the model. In a function-call (export-function) model, Function Call Subsystem blocks that specify a sampling rate represent periodic functions.

MultiRateComponent_Periodic

Exported function: <Root>/Periodic. Must be called periodically, every 0.001 seconds.
#include "MultiRateComponent.h" void MultiRateComponent_Periodic(void)
Calls services:
  • Receiver: get_MultiRateComponent_Periodic_In #include "services.h" real_T get_MultiRateComponent_Periodic_In(void)

    Data communication method: Outside execution
    Target platform communicates data before or after generated code executes.

  • Data transfer: set_MultiRateComponent_Periodic_DataTransferAtF1Outport1 void set_MultiRateComponent_Periodic_DataTransferAtF1Outport1(real_T aVal)

    Data communication method: Outside execution
    Target platform communicates data before or after generated code executes.

3.4 Timer Service Interfaces

Entry-point functions get the function clock tick for a platform environment by calling timer service functions. A call to a timer service is represented in a model as a Discrete Time Integrator or Weighted Sample Time block.

No timer service interfaces represented in model.

3.5 Parameter Tuning Service Interfaces

Parameter tuning service interfaces provide access to parameters that can be tuned at run time. A parameter service interface is represented in a model as a block parameter or a model argument.

No parameter tuning service interfaces represented in model.

3.6 Measurement Service Interfaces

Measurement service interfaces provide access to the internal state of the generated code. A measurement service interface is represented in a model as a signal line or a block with internal state, such as an Integrator block.
